01 / ARCHITECTURE
One request, four layers
Every order travels the same path — CLI parsing is kept completely separate from validation, execution, and the API client.
01

CLI

Parses --symbol --side --type --quantity --price with argparse.

cli.py
02

Validators

Rejects bad input before any network call — symbol format, side, type, quantity, price rules.

bot/validators.py
03

Executor

Builds the request payload, calls the client, normalizes the response.

bot/orders.py
04

Client

Talks to the Futures Testnet only, translates errors into friendly exceptions.

bot/client.py
↳ throughout: every request payload, endpoint, response and exception is written to logs/market_order.log or logs/limit_order.log via bot/logging_config.py
02 / VALIDATION
Nothing reaches Binance unvalidated
Six checks run locally, in order, before a single HTTP request is made.
✓
Symbol format

Must match a USDT-M pair, e.g. BTCUSDT — otherwise rejected.

✓
Side

Only BUY or SELL are accepted, case-insensitive.

✓
Order type

Only MARKET or LIMIT are accepted.

✓
Quantity

Must be a positive number — zero or negative is rejected.

✓
LIMIT requires price

A LIMIT order without --price fails validation immediately.

✓
MARKET rejects price

Passing --price with a MARKET order is a validation error.
